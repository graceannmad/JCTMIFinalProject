# JCTMI Final Project — Claude Code Context

<!-- See AGENTS.md for Next.js version notes before editing any Next.js code -->

## Project Overview

Final project for **Jewish Culture in Translation in Medieval Iberia (JCTMI)**.

A web app where users take a short quiz about their music taste and get matched with a medieval Iberian love poet. The app generates a custom playlist "curated" by a fictionalized version of that poet, written in their voice and sensibility.

**Class context:** Academic project. Demo Wednesday 2026-04-29. Graded deliverable. Site stays live through May.

## Tech Stack

- **Language:** TypeScript (strict mode)
- **Framework:** Next.js 16 (App Router) — frontend + API routes in one deployment
- **Styling:** Tailwind CSS v4
- **AI:** Anthropic Claude API (`claude-sonnet-4-6`) — matching, playlist generation, poet-voice narration
- **Lyrics:** Genius API (server-side proxy via `/api/lyrics`)
- **Song verification:** Spotify Web API (client credentials flow — track search + URL retrieval)
- **Corpus storage:** `/data/corpus.json` bundled in repo (context-stuffed into Claude — no vector DB at 25-page scale)
- **OCR:** Claude Vision API (offline script at `/scripts/ocr.ts` — run before deploy when adding scans)
- **Hosting:** Vercel (free tier, auto-deploy from GitHub on push to `main`)

**Run locally:** `npm run dev`
**Build:** `npm run build`
**Deploy:** push to `main` → Vercel auto-deploys

**Environment variables** (set in Vercel dashboard + `.env.local` for dev):
- `ANTHROPIC_API_KEY`
- `GENIUS_API_KEY`
- `SPOTIFY_CLIENT_ID`
- `SPOTIFY_CLIENT_SECRET`

## Project Structure

```
/app
  /api
    /health        GET  — liveness check
    /lyrics        POST — Genius API proxy
    /generate      POST — full AI pipeline (matching + playlist + Spotify verification)
  layout.tsx
  page.tsx         — landing page + quiz flow (Phase 3)
  globals.css
/data
  corpus.json      — OCR'd poet text chunks (generated by /scripts/ocr.ts)
  poets.json       — hand-authored poet profiles + voice descriptions
/fixtures
  demo-result.json — hand-authored ResultsPayload for bypass mode
  /lyrics          — pre-cached lyrics for demo day rate-limit resilience
/corpus
  /raw             — scanned PDF/image inputs for OCR
  /reviewed        — OCR output to review before committing
/scripts
  ocr.ts           — Claude Vision OCR pipeline (run offline, not in live app)
/lib
  geniusClient.ts
  spotifyClient.ts
  claudeClient.ts
/docs              — PRD, architecture, planning docs
```

## Dev Bypass Mode

Add `?bypass=true` to any URL to skip all external API calls and render fixture data from `/fixtures/demo-result.json`. Use during UI development to avoid burning API quota.

## Key Decisions

ADRs in `docs/architecture/adr/`:
- **ADR-001:** Context stuffing over ChromaDB (25 pages fits in Claude context window; ChromaDB can't run on Vercel serverless)
- **ADR-002:** Spotify track verification as hallucination guard (every displayed song is a verified real track)
- **ADR-003:** Claude Vision for OCR (handles variable scan layouts without special-casing)
